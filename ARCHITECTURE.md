# Архитектура решения

## Цель

Сделать безопасный и повторяемый способ заменить локальные папки `skills` у нескольких AI-agent инструментов на единый источник навыков:

- без сканирования всего диска;
- только по заранее известным и разрешенным путям;
- с резервной копией локальных данных;
- с возможностью повторного запуска;
- с предсказуемым откатом.

## Базовая модель

Источник навыков задается один раз параметром `SourcePath` и должен указывать на корень `skills`.

Этот источник считается единственным master-источником. Все поддерживаемые инструменты получают ссылку на него вместо своей локальной папки `skills`.

## Принятые решения

- Использовать только allowlist известных мест установки.
- Не делать глобальный поиск по диску.
- Для Windows использовать `Junction` как основной тип ссылки, чтобы не требовать административных прав.
- Хранить метаданные установки рядом с резервной копией.
- Считать операцию успешной только если локальная `skills` действительно указывает на переданный источник.
- Делать rollback только для путей, которые были обработаны этим решением.

## Allowlist мест установки

Скрипты работают только по этим корневым папкам `skills`:

- `~/.claude/skills`
- `~/.windsurf/skills`
- `~/.codex/skills`
- `~/.github/skills`
- `~/.gemini/skills`
- `~/.opencode/skills`
- `~/.gemini/antigravity/skills`

Важно: речь идет о корне `skills`, а не о вложенных подпапках отдельных навыков.

## Данные и состояние

Для каждого обработанного места нужен локальный маркер состояния внутри `skills-backup`.

Схема хранения:

- `skills-backup` хранит исходное содержимое локальной папки;
- `.skills-unifier/state.json` хранит метаданные установки;
- при конфликте с уже существующим `skills-backup` используется snapshot-режим внутри `skills-backup/.skills-unifier/snapshots`.

Marker-файл фиксирует:

- исходный путь;
- целевой source path;
- тип ссылки;
- режим хранения backup;
- дату создания и обновления.

## Поток установки

1. Принять `SourcePath` как обязательный параметр.
2. Нормализовать путь источника и убедиться, что он существует.
3. Загрузить allowlist известных мест установки из `config/known-locations.psd1`.
4. Для каждого пути из allowlist:
   - проверить, существует ли родительский каталог;
   - если путь `skills` отсутствует, создать ссылку;
   - если `skills` уже является ссылкой на тот же источник, ничего не делать;
   - если `skills` является обычной директорией, переместить ее в `skills-backup`;
   - если `skills-backup` уже занят, сохранить снимок в `skills-backup/.skills-unifier/snapshots`;
   - создать `Junction` на `SourcePath`;
   - записать marker-файл;
   - вывести понятный лог.
5. Сформировать итоговую сводку: что найдено, что пропущено, что изменено, что восстановлено.

## Поток отката

1. Прочитать marker-файл и убедиться, что каталог управляется этим решением.
2. Для каждого управляемого места:
   - удалить link `skills`;
   - восстановить папку из `skills-backup` или snapshot-режима;
   - по умолчанию сохранить backup;
   - удалять backup только при явном флаге;
   - вывести понятный лог.
3. Если marker-файл отсутствует, не трогать каталог.

## Повторный запуск

Повторный запуск должен быть безопасным и идемпотентным:

- уже настроенные места пропускаются;
- новые известные места обрабатываются автоматически;
- смена источника обновляет существующие ссылки;
- существующий marker-файл используется как признак владельца состояния;
- неизвестные каталоги не затрагиваются.

## Правила безопасности

- Не перезаписывать чужой `skills-backup` без явного подтверждения.
- Не удалять пользовательские файлы без резервной копии.
- Не менять пути вне allowlist.
- Не использовать рекурсивный поиск по файловой системе.
- Не требовать elevated rights как обязательное условие.
- Не считать операцию завершенной, пока link не указывает на правильный source.

## Тестовый контур

Перед реальными каталогами нужна проверка на безопасных тестовых папках:

- тестовая установка;
- повторный запуск;
- смена source;
- rollback;
- конфликт с существующим `skills-backup`.
