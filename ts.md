# Техническое задание: универсальный установщик и откатчик `skills`

## 1. Роль
Ты - senior Windows automation engineer / DevOps-архитектор, который глубоко понимает PowerShell, NTFS-ссылки, структуру пользовательских профилей Windows, идемпотентные установщики, безопасный rollback и автоматизацию для CLI-инструментов.

## 2. Суть
Нужно создать набор PowerShell-скриптов, который переводит папку `skills` у нескольких AI-agent инструментов на единый источник навыков. Решение должно безопасно сохранить существующие локальные данные, заменить папки ссылками на общий источник, поддерживать повторный запуск и уметь откатывать изменения назад без потери файлов.

Цель - получить предсказуемый, повторяемый и безопасный способ централизованного управления `skills` без ручного копирования между инструментами.

## 3. Задача
- [ ] Зафиксировать список известных и разрешенных мест, где у поддерживаемых AI-agent инструментов может находиться папка `skills`.
  - [ ] Claude Code
  - [ ] Windsurf
  - [ ] Codex
  - [ ] GitHub Copilot
  - [ ] Gemini CLI
  - [ ] OpenCode
  - [ ] Antigravity
- [ ] Спроектировать универсальный установочный скрипт, который принимает путь к исходной папке `skills` как обязательный параметр и не хранит этот путь внутри себя.
- [ ] Для каждого известного места установки выполнить единый безопасный сценарий:
  - [ ] проверить, существует ли родительский каталог и разрешено ли это место согласно списку;
  - [ ] проверить наличие `skills`;
  - [ ] если `skills` уже является корректной ссылкой на переданный источник, ничего не менять;
  - [ ] если `skills` является обычной директорией, сохранить ее содержимое в `skills-backup` рядом с ней;
  - [ ] если `skills-backup` уже существует, не затирать его молча и действовать по явно описанному безопасному правилу;
  - [ ] создать или обновить ссылку `skills` на переданный источник;
  - [ ] вывести понятный лог: найдено, изменено, пропущено, восстановлено;
  - [ ] не трогать никакие каталоги вне allowlist.
- [ ] Обеспечить идемпотентность:
  - [ ] повторный запуск не должен ломать уже обработанные пути;
  - [ ] новые известные места должны обрабатываться при следующем запуске;
  - [ ] если источник `skills` меняется, ссылки должны корректно обновляться.
- [ ] Сделать тонкую локальную обертку для конкретного source path и хранить ее в `private/` (gitignored), не публикуя в public repo.
- [ ] Сделать отдельный скрипт отката:
  - [ ] удалить созданные ссылки;
  - [ ] восстановить локальные папки из `skills-backup`;
  - [ ] по умолчанию сохранить резервные копии, а удалять их только по явному правилу или флагу;
  - [ ] быть безопасным при повторном запуске;
  - [ ] не затрагивать места, которые не были установлены этим решением.
- [ ] Подготовить проверку на безобидных тестовых папках до применения на реальных каталогах:
  - [ ] проверить установку;
  - [ ] проверить повторный запуск;
  - [ ] проверить смену источника;
  - [ ] проверить откат.
- [ ] Подготовить финальную документацию по использованию, ограничениям, поддерживаемым путям и порядку запуска.

## 4. Контекст
- Решение предназначено только для Windows-окружения.
- Основной источник навыков всегда передается параметром `SourcePath` в универсальный скрипт.
- Скрипты должны работать только по известным местам установки, а не сканировать весь диск.
- В некоторых местах `skills` уже может быть ссылкой, а в некоторых - обычной директорией с данными.
- Каталоги разных инструментов могут находиться в пользовательском профиле и скрытых системных папках.
- Локальные персональные wrappers можно хранить в `private/`, который не публикуется.
- Приоритет - безопасность пользовательских данных, предсказуемость поведения и удобство повторного запуска.
- Решение должно оставаться понятным для дальнейшей поддержки и расширения списка мест установки.

## 5. Формат
- Итог должен включать минимум 3 PowerShell-скрипта:
  - универсальный установщик;
  - локальный wrapper, если он нужен;
  - rollback-скрипт.
- Скрипты должны быть пригодны к запуску из PowerShell на Windows.
- Желательно использовать явные параметры, понятные коды выхода и логирование в stdout без лишнего шума.
- Желательно поддержать безопасный режим проверки, например `-WhatIf` и `-Confirm`, если это не усложняет сценарий.
- Для каждого скрипта нужна ясная документация:
  - что он делает;
  - как его запускать;
  - что считается успешным результатом;
  - как выполняется откат.

## 6. Ограничения
- Нельзя менять пути вне списка известных и явно поддерживаемых локаций.
- Нельзя делать необратимые действия без резервной копии.
- Нельзя хардкодить source path внутри универсального скрипта - он должен передаваться параметром.
- Нельзя ломать уже существующие корректные ссылки.
- Нельзя молча перезаписывать уже существующий `skills-backup`.
- Нельзя удалять пользовательские данные без явного и безопасного механизма восстановления.
- Нельзя рассчитывать на одноразовый запуск - решение должно быть идемпотентным.
- Нельзя завершать работу без проверки на тестовых, безопасных папках.
- Нельзя публиковать в public repo персональные wrappers с жестко заданным source path.
- Нельзя полагаться на административные привилегии, если их можно избежать.
